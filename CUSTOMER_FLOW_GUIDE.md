# Полная инструкция для заказчика: функционал бота

## 1) Общий сценарий работы (end-to-end)
1. Сотрудник запускает бота (`/start`) и проходит регистрацию.
2. После подтверждения карточки заявка уходит юристу.
3. Юрист отправляет сотруднику файл NDA.
4. Сотрудник загружает подписанный NDA.
5. Юрист получает подписанный файл, проверяет и подтверждает NDA.
6. Пользователь активируется и получает доступ к рабочим действиям.
7. Далее сотрудник может:
   - сдавать факт выполненных работ;
   - создавать заявки на оплату;
   - загружать и отслеживать документы.

---

## 2) Полный функционал по ролям

### Сотрудник
- Запуск и регистрация через `/start`.
- Заполнение карточки контрагента:
  - ФИО;
  - паспортные данные;
  - дата рождения;
  - адрес регистрации;
  - ИНН;
  - телефон;
  - email;
  - дата начала работы;
  - тип налогообложения;
  - документ по налогу (для нужных режимов);
  - выбор отдела.
- Подтверждение введенных данных перед отправкой.
- Получение NDA от юриста.
- Загрузка подписанного NDA.
- Запрос продления NDA (если нужен перенос срока).
- Создание заявки на оплату (счёт):
  - сумма;
  - назначение платежа;
  - контрагент;
  - проект;
  - номер договора;
  - загрузка счёта.
- Просмотр своих заявок.
- Загрузка документов к заявке (договор/акт/чек/подтверждение оплаты в зависимости от этапа).
- Закрытие заявки после загрузки обязательных документов.
- Сдача факта выполненных работ (ежемесячный отчёт):
  - описание;
  - сумма;
  - реквизиты (если не заполнены ранее).
- Управление банковскими реквизитами.
- Отправка уведомления об изменениях.
- Просмотр раздела «Мои документы».

### Юрист
- Просмотр новых запросов на NDA.
- Просмотр карточки сотрудника из запроса.
- Отправка NDA сотруднику.
- Получение подписанного NDA от сотрудника (именно файлом).
- Проверка подписанного NDA:
  - принять NDA;
  - вернуть на доработку.
- Активация пользователя после подтверждения NDA.
- Обработка заявок на оплату на своём этапе.
- Кнопки/разделы:
  - «Подписанные документы» (текущий статус: заглушка);
  - «Чеки» (текущий статус: заглушка);
  - «Выбрать отдел».

### Руководитель
- Выбор отдела для работы.
- Просмотр отчётов сотрудников по отделу.
- Проверка отчётов:
  - подтверждение;
  - отправка на корректировку.
- Просмотр заявок на оплату по отделу.
- Изменение должности сотрудника.
- Увольнение/блокировка сотрудника (в рамках доступных действий раздела).

### Финансы / Бухгалтерия
- Просмотр заявок на оплату.
- Работа с платёжными поручениями.
- Подтверждение оплаты.
- Раздел «Заявки в ожидании оплаты».
- Связь с сотрудником по заявке.

### Админ
- Управление пользователями.
- Добавление пользователя.
- Изменение роли пользователя.
- Блокировка пользователя.
- Просмотр статистики.
- Сервисная команда очистки пользователей (`/clear_users`).

---

## 3) Статусы и логика NDA
- После отправки NDA сотруднику: статус NDA меняется на «отправлен».
- После загрузки подписанного файла сотрудником: файл сохраняется и уходит юристу на проверку.
- После подтверждения юристом: NDA подтвержден, пользователь активирован.
- Уведомления разделены:
  - проверяющему — короткое подтверждение;
  - сотруднику — полное сообщение с дальнейшими шагами.

---

## 4) Ограничения доступа
- Большинство действий доступны только пользователям с нужной ролью.
- Действия сотрудника после регистрации доступны только активированному пользователю.
- Пока NDA не подтвержден, доступ к части рабочих действий ограничен.

---

## 5) Текущее состояние реализации
- Основные бизнес-процессы (регистрация, NDA, отчёты, заявки на оплату) реализованы.
- Некоторые разделы у юриста помечены как заглушки и могут быть доработаны отдельно.
